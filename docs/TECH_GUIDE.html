Chapter 1

1 · Executive Summary & System Overview

MCQ Café POS is a complete, two-faced point-of-sale system — a cashier register and a self-service kiosk — that runs entirely inside a web browser with no server-side application logic and no build step. It is written in framework-free ("vanilla") HTML, CSS and JavaScript.

The product serves a single Vietnamese street-food café. One menu, one price book and one sales history are shared between two front-ends:

  • The Register — staff till: ordering, customisation, split payments, receipts, refunds/voids, reporting, menu & settings administration.
  • The Standing Order Tree — a customer-facing self-order kiosk: attract loop, bilingual (EN/VI) menu, upsell, loyalty and queue tickets. Kiosk sales drop into the same history the cashier reads.

At a glance

PropertyValue
LanguagesHTML5, CSS3, ES2017+ JavaScript
Frameworks / build toolsNone. No React/Vue, no bundler, no transpiler, no npm runtime dependency
Runtime dependenciesGoogle Fonts (CDN, optional) — everything else is local
PersistencelocalStorage (business data) + IndexedDB (uploaded media blobs)
Source size~7,300 lines across 4 JS files, 1 stylesheet, 1 HTML shell
Back-endNone required. Optional static-file host (e.g. PythonAnywhere/Flask) only serves the files
BrowsersChrome / Edge (recommended), Safari, Firefox — desktop & touch
Why "no backend"?
The app keeps all state on the device. That makes it trivial to host (any static server or even file://), resilient offline, and free of database/operations overhead — at the cost of being single-device. Chapter 16 describes the path to a multi-terminal, server-backed deployment when the business needs it.

Chapter 3

3 · Technology Stack & Design Methodology

The guiding principle is radical simplicity and longevity: a system any developer can read end-to-end, that has no dependency tree to rot, and that will still run unchanged in a browser ten years from now.

Why framework-free?

  • Zero build / zero install. Edit a file, refresh the browser. No node_modules, no webpack, no version drift.
  • No supply-chain surface. Nothing to npm audit; no transitive packages to patch.
  • Full control of performance. Rendering is explicit, so hot paths (cart, kiosk) update surgically without a virtual-DOM diff (Chapter 13).
  • Longevity. The platform is the standard library — DOM, localStorage, IndexedDB, <video>.

Patterns & methods used

PatternWhere & why
Module pattern (IIFE)Every file is (function(global){ … })(window) — private scope, one explicit export. Replaces ES modules without a bundler.
Single source of truthOne state object in store.js holds everything; the UI is a pure function of it.
Observer / pub-subStore.subscribe(fn) + commit(). Any mutation calls commit(), which persists and notifies the UI to re-render.
Declarative views, imperative patchesViews are template-string render functions; high-frequency updates patch only the changed sub-tree.
Finite state machineThe kiosk (attract → home → menu → member → pay → done) and the payment flow are explicit stages.
Schema versioning & migrationseedVersion on the config drives forward-migration of persisted data (Chapter 5).
Graceful degradationPersistence, media storage and autoplay all wrapped in try/catch and feature checks so a blocked API never breaks the app.

Domain modelling method

The POS domain is modelled as plain data with pure transforms. An order is a list of lines; a line references a menu item plus chosen add-ons, modifiers, note and quantity. Pricing is a pure function computeTotals(cart) — no side effects, fully unit-testable. Payment, refund and promotion rules are likewise expressed as data-in/data-out functions over that model.

Convention
Money is integer-safe via round2(); identifiers are short slugs (banh-mi, u-admin); time buckets use dayKey(ts) for day-grouped reports.

Chapter 5

5 · The Data Layer & "Database"

There is no SQL database and no server. The persistence tier is two browser storage APIs wrapped by store.js and media.js.

5.1 Business data — localStorage

The entire live business state is serialised to a single localStorage key. Reads/writes are wrapped so a blocked storage API (private mode) degrades to in-memory only.

store.js
const KEY = 'mcq_pos_state_v1';

function persist(){ try{ localStorage.setItem(KEY, JSON.stringify(state)); }catch(e){} }
function commit(){ persist(); emit(); }              // mutate → save → notify UI
function subscribe(fn){ listeners.add(fn); return () => listeners.delete(fn); }

The persisted object (the "schema") is a single JSON document:

KeyShapeNotes
configstore details, tax, logo, users, flags, seedVersionEditable in Settings; also holds media flags (Chapter 10)
menuarray of items (price, availability, category, image)Seeded from MCQ_DATA.MENU, then user-editable
promotionsdate-aware discount rulesAuto activate/expire by date
cartthe in-progress order (lines, type, discounts)The live register order
orderscompleted sales (newest first)Shared by counter + kiosk; queried for receipts & refunds
heldparked ticketsRecall/void
auditLogsensitive-event trailSales, refunds, voids, discounts, settings changes
credits / tillsgoodwill credits, till readingsCash management
livecustomer-display channelMirrors the active cart to the second screen

5.2 Schema migration

On load(), a stored seedVersion below the current (MCQ_DATA.CONFIG.seedVersion, presently 12) triggers forward-migration — adopting new default fields without clobbering user edits. This is how the codebase ships breaking dataset changes safely to devices that already hold data.

store.js · migrateConfig()
const savedVer = savedConfig.seedVersion || 1;
if (savedVer >= (D.CONFIG.seedVersion || 1)) return;   // already current
// …adopt new defaults only where the user never customised…
state.config.seedVersion = D.CONFIG.seedVersion;

5.3 Uploaded media — IndexedDB

Images/videos uploaded by an admin are far too large for localStorage (~5 MB cap), so media.js stores them as Blobs in IndexedDB. Synchronous render code can't await IndexedDB, so the module keeps an in-memory cache of Object URLs built once at boot by preload().

media.js · object store
DB "mcq-media" → object store "media" { keyPath:"id", index:"slot" }
record = { id, slot, name, type, kind, size, ord, blob, createdAt }
slots  = "ad" | "login" | "customer"   // each a rotating list

cache  = { ad:[…urls], login:[…urls], customer:[…urls] }  // rebuilt on every add/remove
Why object URLs
Blobs are turned into blob: URLs once and reused, so a <video src> streams straight from memory. Old URLs are revokeObjectURL'd on each rebuild to avoid leaks. See Chapter 10 for the full subsystem.
Chapter 6

6 · State Management & the Rendering Model

The UI is a function of state. A mutation calls Store.commit(); the store persists and emits; the UI re-renders. The art is re-rendering only what changed.

The render pipeline

ui.js
function init(){
  Store.load();
  // build #view / #modal-root / #print-area / #toast shells
  Store.subscribe(renderDynamic);       // re-render on every commit
  render();
  if (Media) Media.preload().then(repaintMediaViews);   // swap in uploaded media when ready
}

function renderDynamic(){
  // SURGICAL fast-paths — avoid blowing away the whole view:
  if (ui.view === 'register' && cartEl) return refreshCart();   // only #pos-cart
  if (ui.view === 'backscreen') return paintBackScreen();    // only the order half
  if (ui.view === 'kiosk') return;                       // kiosk drives its own paints
  renderTopbar(); render();                                // otherwise full repaint
}

Three tiers of update

TierMechanismUsed by
Fullrender() rewrites #view.innerHTML from a template stringNavigation between views
SurgicalReplace only one sub-element (e.g. #pos-cart, .cust2-order, a kiosk card)High-frequency state (cart edits, kiosk +/−)
Out-of-bandModals in #modal-root; receipts in #print-area; toastsDialogs, printing, transient messages

For example, adding an item at the register never rebuilds the 70-item menu grid — only the cart panel is re-rendered, preserving its scroll position. On the kiosk, the stepper updates a single card and the bottom bar in place. This is what keeps the touch UI flicker-free (Chapter 13).

Event wiring
There is no event-delegation framework: markup uses inline onclick="UI.fn(args)". Every handler is therefore an explicit member of the UI export object, which doubles as the public surface for tests.
Chapter 7

7 · Core Domain Logic

All pricing and fulfilment rules live in store.js as pure-ish functions over the data model. This chapter maps the most important ones.

Cart & line construction

buildLine(item, opts) composes a cart line from a menu item plus chosen add-ons, modifiers, note and quantity. addItem/setQty/incQty/updateLine/removeLine mutate the cart and commit().

Totals & tax

computeTotals(cart) is the single pricing authority: it sums lines, applies per-line and order-level discounts, resolves active promotions, computes tax and (optional) cash rounding, and returns a totals object. Being pure, it is the most heavily unit-tested function in the system.

Payments

  • Split tender across Cash / Card / Other, each partial amount tracked until the balance is cleared.
  • Cash keypad with automatic change calculation and optional rounding.
  • Card (EFTPOS) is a timed simulation — no real processor. Replacing this is the #1 production task (Chapter 16).

Completing, refunding & voiding

store.js — order lifecycle
completeOrder(payment)   // cart → orders.unshift(order); assigns daily queue no.; audits; clears cart
findOrder(id) · searchOrders(q)   // receipt lookup by id / queue / amount / name
refundOrder(id, lines) · voidOrder(id)
refundableQty(...) · remainingRefundable(...)   // partial-refund guards

Promotions

Promotions are date-aware: promotionActiveNow() / promotionStatus() switch rules on and off automatically by date, so seasonal offers need no manual toggling.

Shared daily queue number

Key integration detail
The counter and the kiosk draw from the same daily sequence. A kiosk sale is orders.unshift'd into the very same history list, so the cashier can look it up, reprint its receipt, or refund it — there is no separate "kiosk orders" silo.

Loyalty

getMembers / findMember / addMemberPoints back a simple points scheme (seed members MCQ1001–1003), surfaced at both the till and the kiosk.

Chapter 9

9 · The Self-Service Kiosk

The Standing Order Tree is a full-screen state machine with its own cart held in UI state (not the store), an idle auto-reset, and bilingual copy.

Stages

ui.js — kiosk FSM
attract → home → menu (+cart) → member → pay → done(queue ticket)
ui.kioskStage drives renderKiosk(); kioskPaint() rewrites #view directly
armKioskIdle()  // inactivity → wipe cart, return to attract (abandoned-order safety)

Internationalisation (EN / VI)

A tiny i18n layer — kt(key) over a KT_DICT dictionary — localises kiosk copy; a header toggle flips ui.kioskLang and repaints.

Surgical kiosk updates

Each menu card carries a data-id and an always-present quantity badge. A tap routes through kioskSyncCard(id) + kioskSyncBar() which update just that card and the checkout bar — so no image ever reloads and the grid never rebuilds.

Attract advertising & neon frame

The attract screen plays the ad playlist (Chapter 10) inside a CSS-only glowing neon frame with hanging lanterns and corner lights. The frame animates opacity only, so it composites cheaply.

Chapter 10

10 · The Media Subsystem

Three admin "media managers" let staff run their own photos/videos on the kiosk ad loop, the login screen and the customer display — with no back-end. This is the newest subsystem; IT should understand it well.

Storage model (media.js)

One IndexedDB object store, three logical slots, each an ordered list. Uploading appends; the in-memory cache of object URLs is rebuilt after every change so synchronous render code stays simple.

media.js — public API
Media.preload()                // boot: blobs → object-URL cache
Media.add(slot, file)          // store a Blob, rebuild cache
Media.remove(id) · clearSlot(slot)
Media.ads() · login() · customer()   // synchronous getters → [{id,name,kind,size,url}]

Built-in + uploaded, the same logic on all three screens

Each manager presents one unified list:

  • Built-in defaults (the clips shipped in assets/videos/) appear as real, removable rows. "Removing" a built-in records its id in a config flag — it is not deleted from disk.
  • Uploads are appended and stored in IndexedDB.
  • When a screen holds more than one item, items rotate — images on a timer, videos on onended.
  • A Restore action clears the hidden-defaults flag.
ui.js — the "active list" pattern (ads shown)
const DEFAULT_ADS = [{ id:'def-ad1', url:'assets/videos/ad1.mp4?v=2', builtin:true }, … ];

function activeAds(){
  const hidden = Store.getConfig().kioskAdsHiddenDefaults || [];
  return DEFAULT_ADS.filter(d => !hidden.includes(d.id))
              .concat(Media.ads());          // built-ins (kept) + uploads
}
ScreenSlotBuilt-inConfig flag(s)
Kiosk attractadad1.mp4, ad2.mp4 (HD)kioskAdsHiddenDefaults, kioskAdsOff
Login screenlogin3-panel collageloginHiddenDefault, loginVideoOff
Customer displaycustomervid-847.mp4customerHiddenDefault
Operational consequence
Media lives in the browser of the device that uploaded it. There is no sync across machines — each terminal manages its own content. A future CMS would centralise this (Chapter 16).
Chapter 11

11 · Security & Access Control

Access control is role- and PIN-based, enforced in the client. It is appropriate for an in-store, single-tenant till — but it is not a server-grade security boundary, and the document is explicit about that.

Roles & gating

AccountPIN (default)Role
Manager1234admin
Cashier 11111cashier
Cashier 22222cashier
  • GATED_VIEWS (admin, dashboard, menu, settings, audit) require a manager; render() bounces a cashier back to the register.
  • requireAdmin(cb) unlocks admin features once per shift on a manager PIN; requireAdminAlways(cb) re-prompts every time (e.g. discounts).
  • Transaction History is open to all; only the refund action inside it is gated.
Threat model — be honest with stakeholders
PINs and data live in the browser. This protects against casual misuse by staff, not a determined attacker with device access (data is readable in DevTools / localStorage). Card payments are simulated. For handling real money and PII at scale, move auth and payments server-side (Chapter 16). Change the default PINs before going live.

12 · Testing & Quality Assurance

Because the model is decoupled from the DOM, it is tested two ways: static checks, and headless-browser behavioural smoke tests driving the real UI surface.

Static & syntax

Every change is validated with node --check on each JS file (parse-level guarantee) before it is run in a browser.

Headless behavioural tests (puppeteer-core + system Chrome)

A real Chrome loads the app from a local static server and exercises features through the public window.UI / window.Media / window.Store globals — the same surface the markup calls. Assertions check DOM outcomes and that the console is error-free.

example smoke test (abridged)
await page.evaluate(() => { UI.chooseCashier('u-admin'); ['1','2','3','4'].forEach(UI.pinKey); });
await page.evaluate(async () => { await Media.add('ad', file); await Media.preload(); });
// assert: ad rows rendered, blob URLs, rotation advances, zero console errors

Representative coverage already exercised: media add/remove/restore across all three slots, rotation, login PIN entry & the video-pause performance fix, kiosk surgical add/remove, and the customer-display rotation.

Mocked-DOM unit tests (Node vm)

Domain functions (computeTotals, order lifecycle) can be loaded into a Node vm with a minimal DOM mock for fast, browser-free assertions.

Recommended next step
Commit the smoke tests into a test/ folder and wire a package.json script + GitHub Action so they run on every push. The harness already exists; it just needs to live in the repo.

On touch hardware, perceived smoothness is everything. The codebase applies a consistent set of techniques to keep frames cheap.

Stable media elements — keep a playing <video> alive across data updatesCustomer display order repaint

Chapter 14

14 · Build, Deployment & Operations

There is nothing to build. "Deploying" means serving the static files; "releasing" means bumping the cache-buster.

Local development

# any static server from the repo root
python3 -m http.server 8080      # → http://localhost:8080

Edit a file, hard-refresh the browser. No watch process, no compile.

Hosting

  • Any static host works: PythonAnywhere (Flask serving the folder + a static-files mapping), Netlify, Cloudflare Pages, GitHub Pages, S3+CloudFront, or a plain nginx.
  • For media-heavy smoothness, serve /assets/ through a CDN / static mapping rather than the app process.

Release procedure

  1. Make and test changes (node --check + headless smoke).
  2. Bump ?v=N on the CSS/JS tags in index.html (current: 40).
  3. Commit, push, redeploy the static files, reload the host.
  4. Hard-refresh open devices once.

Backup & restore

Settings → Export/Import writes/reads the whole localStorage document as JSON. This is the operational backup path per device. Uploaded media (IndexedDB) is not included in that JSON and must be re-uploaded if a device is wiped.

Chapter 15

15 · Extending the System

Common change recipes, with the exact touch-points.

Add a menu item

Add to MCQ_DATA.MENU in data.js (id, name, price, category, image) and bump seedVersion if existing devices must adopt it — or add it at runtime via Menu Manager. Item photos live in assets/images/items/<id>.jpg.

Add a new screen/view

  1. Write renderXyz() returning a template string.
  2. Add a branch in render() on ui.view === 'xyz'.
  3. Expose any handlers on the UI export object; reference them as onclick="UI.fn()".
  4. If it is manager-only, add it to GATED_VIEWS.

Add a new media slot

  1. In media.js: add the slot to cache and the preload/revokeAll branches; add a getter.
  2. In ui.js: define defaults + an active<Slot>() list, a manager modal mirroring the existing three, and a rotation arm if needed.
  3. Add a tile in renderAdminHub and export the handlers.

Add a business rule (e.g. a tax band, a promotion type)

Implement it as a pure function in store.js and route it through computeTotals; add a unit test. Keep all pricing in one place.

House style
Match the surrounding code: 2-space indent, single quotes, terse helpers, comments that explain why. New markup uses inline UI.* handlers; new state changes end in commit().
Chapter 16

16 · Known Limitations & Production Roadmap

The architecture is deliberately single-device and client-only. These are the conscious trade-offs and the upgrade path when the business outgrows them.

Current limitations

LimitationImpact
Data is per-device (localStorage)No shared history across terminals; backup is manual JSON export
Media is per-device (IndexedDB)Each terminal must be configured separately
Card payments are simulatedNo real settlement
Client-side authNot a hard security boundary; data visible in the browser
No real-time multi-terminal syncTwo tills can't share one live order book

Recommended path to production

  1. Back-end API + database (e.g. Node/Python + Postgres) as the source of truth for menu, orders, members and audit. The current Store becomes a thin client over an HTTP/realtime API — its function surface barely changes.
  2. Real payments via a PSP (Stripe/Adyen/Square) and/or a hardware terminal, replacing the EFTPOS simulation.
  3. Server-side auth (hashed PINs/credentials, sessions, per-action authorisation) replacing client gating.
  4. Central media CMS — move the three media managers' content to the server so head office pushes ads/branding to all kiosks, with scheduling (dayparting).
  5. Multi-terminal sync (websockets) for a shared live order book and instant menu/price propagation.
  6. CI — repo-committed tests on every push (the harness already exists).
Migration is incremental
Because the model/view split is clean, each step above can be adopted on its own. The fastest first win is putting orders behind an API so multiple terminals share one sales history.
